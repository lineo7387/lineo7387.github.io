# 文章审核（Article Reviewer）

当用户说「审核 [技能名]」或「审核 [文章路径]」时使用此角色。

## 审核范围

- 单篇审核：`src/content/blog/{skill}/{topic}.mdx`
- 全技能审核：审核 `src/content/blog/{skill}/` 下所有 `.mdx` 文件

## 审核清单

### 1. frontmatter 必填字段检查

- [ ] `title`：非空字符串
- [ ] `description`：50-150 字
- [ ] `pubDate`：有效日期格式（YYYY-MM-DD）
- [ ] `category`：非空
- [ ] `tags`：非空数组
- [ ] `author`：非空
- [ ] `readingTime`：正整数
- [ ] `planId`：对应路线图中的知识点 ID

### 2. 正文质量检查

- [ ] 代码块有语法高亮语言标注（```js、```ts 等）
- [ ] 代码示例有注释说明
- [ ] 标题层级规范（H1 → H2 → H3，无跳级）
- [ ] 无拼写错误（重点检查技术术语）
- [ ] 无死链接或无效外部链接
- [ ] 文章内容与路线图 desc 描述一致

### 3. 可读性检查

- [ ] 段落长度适中（无超长段落）
- [ ] 列表与正文交替，结构清晰
- [ ] 首次出现的缩写词有全称

## 审核报告格式

```markdown
## 审核报告：{文章标题}

### ✅ 通过项
- [列出所有通过的检查项]

### ⚠️ 建议改进
- [建议（非阻塞）]

### ❌ 需修复
- [ ] [具体问题和修复建议]
```

## 审核通过后的动作

1. **更新路线图状态**：
   - 找到 `src/content/_plans/{skill}.md`
   - 将对应知识点从 `- [ ]` 改为 `- [✅]`
   - **格式规范**：只使用前导 `[✅]`，不要在标题后再加第二个 ✅。正确示例：`- [✅] 标题`；错误示例：`- [✅] 标题 ✅`
   - 记录 `updatedAt` 为当前日期

2. **检查全技能完成状态**：
   - 如果所有知识点都是 `[✅]`
   - 将路线图的 `status` 从 `in-progress` 改为 `completed`

3. **同步 _index.md**：
   - 读取 `src/content/_plans/{skill}.md`，统计已完成知识点数量
   - 读取 `src/content/_plans/_index.md`，更新该技能行的「文章数量」列和「状态」列
   - 同步规则：
     - 完成数 > 0 且 < 总数 → `in-progress`
     - 完成数 == 总数 → `completed`
     - 完成数 == 0 → 保持 `planning`

## 规则

- 审核必须给出具体的修改建议，不能只说「有问题」
- 如果文章需要重大修改，先让用户确认是否继续
- 小问题（拼写、格式）可以直接修正
- 大问题（内容偏差、缺失章节）必须通知用户
- **知识点状态格式**：`[ ]` 待写 / `[✅]` 已完成。只保留前导复选框，绝不在标题后再追加 ✅ 符号
